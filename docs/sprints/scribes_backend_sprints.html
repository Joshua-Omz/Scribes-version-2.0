Notes, Drafts & Post creation
Content core
The three-stage content pipeline: Notes (private workspace) → Draft (staging) → Post (published artifact). Schema, sqlc queries, and handlers for each.
Schema migrations — notes, drafts, posts, post_categories, scripture_refs, notebooks migrations/002_content.sql · JSONB for content field

sqlc queries + generated Go types for notes and drafts db/query/notes.sql · db/query/drafts.sql · sqlc generate

3 Week 3
Post versioning & Correction Posts
Content core
The versioned transparency model — revise a Post, snapshot the old version immutably, expose Version History. Plus Correction Post as a separate linked artifact.
Schema migration — post_versions (immutable; never updated or deleted) migrations/003_versioning.sql · post_id FK, version_number, content_snapshot JSONB, snapshotted_at
1 pt
PATCH /posts/:id/revise — snapshot current → post_versions, write new content, bump current_version Must verify caller == author. Runs in a single DB transaction.
4 pts
GET /posts/:id/versions — return ordered version history sqlc query on post_versions ORDER BY version_number ASC
2 pts
Correction Post flow — POST /posts/:id/correction Creates new Post row with is_correction=true, corrects_post_id set. Links via erratum footnote in read response.
3 pts
Soft-delete POST — DELETE /posts/:id sets is_deleted=true, returns tombstone on GET Orphan handling: if user later deletes account, handle → [Deleted User]
2 pts
Revision + snapshot in one tx Version history endpoint Correction Post linked Soft-delete tombstone
4 Week 4
Offline sync handler
Sync & offline
The delta sync protocol that powers the local-first client. Server sequence is always the source of truth — client clocks are never trusted.
GET /sync?seq=N — return all notes, drafts, posts WHERE server_sequence > N AND owner = caller Paginated. Returns max_sequence in response so client updates its checkpoint.
4 pts
POST /sync/push — bulk upsert local-only records, assign server_sequence to each via nextval() Returns ack with server_sequence per record. Client stamps local rows.
4 pts
Conflict resolution — last-write-wins by server updated_at (v1 policy) Document the policy in a comment block. v2 deferred.
1 pt
Integration test — simulate offline writes, reconnect, assert delta correctness Test file: internal/sync/sync_test.go
3 pts
GET /sync delta working POST /sync/push ack server_sequence authoritative sync integration test
5 Week 5
Follows, Reactions & Comments
Social layer
The quiet asymmetric follow model, domain-specific reactions, and the flat single-level comment stream with author moderation.
Schema — follows (composite PK, no follower count column), reactions, comments migrations/005_social.sql
1 pt
Follow/unfollow — POST /users/:id/follow · DELETE /users/:id/follow Counts are never exposed in any response. Quiet asymmetric — no approval.
2 pts
Reactions — POST /posts/:id/react {type: AMEN|INSIGHTFUL|THOUGHT_PROVOKING} Public response: aggregate counts. Author response: counts + reactor identities.
3 pts
Comments — POST /posts/:id/comments · parse @mentions into mention_ids[] GET returns flat stream. Author can PATCH is_hidden=true or DELETE.
4 pts
Bookmarks + Imports — POST /saved · type: BOOKMARK or IMPORT IMPORT stores a JSONB content snapshot of the Post at time of saving.
2 pts
Quiet follow model Domain reactions @mention parsing Author moderation on comments
6 Week 6
Feed, Explore & Categories
Discovery
Three feed modes — following feed, explore (interaction-scored), and cold-start — all using only deliberate engagement signals. Taxonomy management baked in.
Schema — categories (name, is_deprecated), user_onboarding_categories migrations/006_feed.sql · only Super Admins can write to categories
1 pt
GET /feed — posts WHERE author_id IN (user follows), reverse-chron, cursor paginated Keyset pagination on (published_at, id). No offset.
3 pts
GET /explore?categories=[] — interaction score = SUM(reactions + comments) rolling 90d No view counts. Score query runs on indexed reactions + comments tables.
4 pts
Cold-start detection — if follow count = 0, feed handler calls explore with onboarding categories Same score formula. Identical analytics guarantees.
2 pts
GET /categories — list active categories for client selectors Returns is_deprecated flag so client can grey out deprecated options
1 pt
Following feed Explore with score Cold-start path Consistent analytics policy
7 Week 7
Direct messaging
Messaging
Mutual-only DMs with message requests, text-only content, soft-delete, blocking, and the full retention/moderation policy from the SRS v2.
Schema — conversations, message_requests, messages (body text only, is_deleted, sent_at) migrations/007_dm.sql · conversation has blocked field
2 pts
POST /messages/request — create pending MessageRequest, notify recipient No message sent until approved. PATCH /messages/request/:id/approve|reject.
3 pts
POST /conversations/:id/messages — send message (mutual check at service layer) Text only. Max length enforced. blocked=true → 403.
2 pts
DELETE /messages/:id — soft-delete, body → "[Message deleted]", sender only POST /conversations/:id/block — sets blocked=true, hides thread from both
2 pts
Retention: background job purges conversations inactive for >12 months Goroutine scheduled job. Logs purge count. Part of next sprint's worker setup.
2 pts
Message request flow Mutual check enforced Soft-delete Block + retention policy

Notifications + Admin
The two-path notification system (real-time for @mentions and admin alerts, batched digest for everything else) plus the Super Admin internal dashboard routes.

2 pts
Admin routes — GET /admin/reports, PATCH /admin/content/:id/action, taxonomy CRUD RequireRole(SUPER_ADMIN) guard on all /admin/* routes
3 pts
GET /admin/categories, POST /admin/categories, PATCH /admin/categories/:id/deprecate Deprecated categories remain on existing Posts. Cannot hard-delete.
2 pts
Real-time @mention path Batch digest scheduler Admin moderation routes Taxonomy management
